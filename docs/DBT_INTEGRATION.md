# dbt Integration & Lineage Detection

Reference for cdar-client-kit's interaction with dbt artifacts and lineage detection.

## Overview

cdar-client-kit Phase 4 integrates with dbt to:

1. **Parse manifest.json** — extract table/model lineage
2. **Detect pre-FHIR transformations** — identify where source → FHIR flattening happens
3. **Emit Tier 3 findings** — when lineage is not instrumented with OpenLineage events
4. **Surface gaps** — where dbt coverage ends and OpenLineage coverage should begin

---

## dbt Manifest Parsing

### Input File

```
dbt_project_root/target/manifest.json
```

Generated by:
```bash
dbt compile
dbt parse
```

### Parsing Strategy

cdar-client-kit reads `manifest.json` and extracts:

1. **Models** — table/view definitions
2. **Sources** — upstream data sources (S3, Aidbox, databases)
3. **Dependencies** — upstream_dependencies, fqn (fully qualified name)
4. **Materialization** — view, table, ephemeral
5. **Meta** — custom dbt meta tags (lineage hints)

### dbt Meta Tags for Lineage

Add metadata to your dbt model YAML to help cdar-client-kit understand lineage:

```yaml
# models/staging/stg_observations.yml
models:
  - name: stg_observations
    description: "Flattened FHIR Observations from source feeds"
    meta:
      dqar_source_feeds:
        - lab-vendor-x-daily
        - ehr-epic-daily
      dqar_source_type: clinical_lab
      dqar_transformation_layer: hl7v2_to_fhir
      dqar_lineage_instrumented: true
    columns:
      - name: observation_id
        description: "FHIR Observation.id"
      - name: source_feed_id
        description: "Feed ID from EXT 6 (ingest-pipeline-id)"
```

## Lineage Detection Validator

### What it checks

Phase 4 lineage detection runs three checks:

#### Check 1: Source → Staging

**Question:** Is there an explicit path from raw source (S3, Aidbox, claims dump) to a staging model?

**Pass criteria:**
- dbt `sources` block references upstream system
- Source has explicit file location or API endpoint
- Materialization is `table` (not ephemeral)

**Tier 3 finding if failed:** "Pre-FHIR Lineage Not Instrumented — No dbt source → staging mapping"

```
Finding: Lab vendor X daily feed has no dbt source definition
Remediation: Add sources.yml entry linking S3 path to dbt source
Rung impact: Cannot move from Rung 2 to Rung 3
```

#### Check 2: Staging → FHIR Flattening

**Question:** Is there a dbt model that explicitly transforms staging → FHIR flat tables?

**Pass criteria:**
- Model exists (e.g., `fct_observations`)
- Upstream models include staging models
- `meta.dqar_transformation_layer` = `hl7v2_to_fhir` or similar
- Model outputs to `main.dqar` schema

**Tier 3 finding if failed:** "Pre-FHIR Lineage Not Instrumented — No HL7v2 → FHIR transform model"

```
Finding: No dbt model transforms raw HL7v2 to FHIR Observation
Remediation: Create dbt model (e.g., fct_observations) with FHIR transformation logic
Rung impact: Cannot demonstrate Tier B compliance
```

#### Check 3: OpenLineage Instrumentation

**Question:** Is lineage instrumented with OpenLineage events at the dbt run level?

**Pass criteria:**
- dbt-openlineage plugin installed
- `dbt run` emits RunEvents to OpenMetadata
- RunEvent includes `CDARIngestFacet` with field mappings
- Field mappings trace HL7v2 OBX segment → FHIR Observation.value

**Tier 3 finding if failed:** "Pre-FHIR Lineage Not Instrumented — dbt runs do not emit OpenLineage events"

```
Finding: dbt transformations are not tracked in OpenMetadata
Remediation: Install dbt-openlineage plugin, configure RunEvent emission
Rung impact: Cannot achieve Rung 3 (Level 3 maturity) without lineage instrumentation
```

## dbt-openlineage Configuration

### Installation

```bash
pip install dbt-openlineage
```

### profiles.yml Configuration

```yaml
your_project:
  target: dev
  outputs:
    dev:
      type: databricks
      # ... other Databricks config ...
      
      # OpenLineage Configuration
      send_raw_sql: true
      
  # OpenLineage event emission
  openlineage:
    transport:
      type: http
      url: "https://your-openmetadata-instance.com/api/v1/events"
      headers:
        Authorization: "Bearer YOUR_TOKEN"
```

### dbt_project.yml Configuration

```yaml
name: 'data_transforms'
version: '1.0.0'

# OpenLineage emit on every run
on-run-start:
  - "{{ openlineage_event() }}"

on-run-end:
  - "{{ openlineage_event() }}"
```

## Finding a Gap: Pre-FHIR Lineage Not Instrumented

### Root Causes

| Cause | Indicator | Fix |
|---|---|---|
| No dbt source defined | dbt parse shows no sources block | Add `sources.yml` with upstream system definition |
| No transform model | No model bridges staging → FHIR | Create dbt model with transform logic |
| dbt-openlineage not installed | dbt run doesn't emit events | `pip install dbt-openlineage` + configure |
| OpenLineage endpoint unreachable | Events fail to emit | Check OpenMetadata URL, auth token |
| No CDARIngestFacet | Events lack field mappings | Add custom facet to dbt RunEvent emit logic |

### Diagnosis Workflow

```bash
# 1. Check dbt manifest for sources
dbt parse
cat target/manifest.json | jq '.sources'

# 2. List models and dependencies
dbt ls --resource-type model --graph

# 3. Check for openlineage plugin
dbt --version
# Should show: "Plugins: openlineage"

# 4. Test OpenLineage emission (dry run)
dbt run --select stg_observations --dry-run --debug

# 5. Verify dqar config in meta
cat models/staging/sources.yml | grep dqar_
```

### Remediation Checklist

- [ ] dbt source defined for upstream system (S3, claims export, etc.)
- [ ] dbt model bridges raw → FHIR flattening
- [ ] dbt-openlineage installed and configured
- [ ] OpenMetadata endpoint URL and auth token set
- [ ] dbt run emits RunEvents to OpenMetadata
- [ ] RunEvents include CDARIngestFacet with field mappings
- [ ] Field mappings trace protocol-specific source fields (HL7v2 OBX, 837 segments, NCPDP fields)

## Best Practices

1. **Always define dbt sources** — even if you're not using dbt for orchestration
2. **Tag transform models** — use `meta.dqar_*` to help cdar-client-kit understand your pipeline
3. **Version your mappings** — dbt model versions should correspond to Interbox rule versions
4. **Emit RunEvents early** — configure OpenLineage before your first dbt run
5. **Include field mappings** — CDARIngestFacet must trace source protocol fields to FHIR target fields
6. **Test emission** — verify events appear in OpenMetadata before certification
